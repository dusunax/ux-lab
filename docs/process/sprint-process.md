## 스프린트 사이클 개요

```
스프린트 회의 (전원 필참)
       |
       v
액션아이템 확정 + 담당 Agent 명시
       |
       v
개발 착수 (담당 FE / BE)
       |
       v
QA 진행 (수용 기준 전항목 검증)
       |
       v
개발자 QA 결과 확인 및 승인
       |
       v
액션아이템 완료 체크박스 업데이트
       |
       v
스프린트 완료 (90% 이상 기준)
       |
       v
PM 보고용 프레젠테이션 작성
```

## 단계별 규칙

### 1. 스프린트 회의

- 개발 시작 전 반드시 스프린트 회의를 먼저 진행한다.
- 참석 대상: **전원 필참** (PM, FE, BE, QA×2, TS, AI)
- 회의 없이 개발을 시작하는 것은 허용하지 않는다.

### 2. 회의록 작성

- 회의록은 **TS(Alex)가 단독으로 일관되게** 작성한다.
- 작성자가 달라지면 형식 일관성이 깨지므로, 부재 시 회의 후 별도 브리핑을 통해 TS가 작성한다.

### 3. 개발 착수 조건

- 회의에서 **액션아이템이 명시되기 전까지 개발을 시작하지 않는다.**
- 회의 전 착수한 코드가 있는 경우: 미완성 상태로 유지하고, 액션아이템 확정 후 재착수한다.
- 확정 전 코드는 PR에 포함하지 않는다.

### 4. 액션아이템 담당 명시

- 모든 액션아이템에 **담당 Agent를 명시**한다.
- 담당자가 명시되지 않은 액션아이템은 착수 불가.
- 담당 외 Agent가 진행한 경우 PM에게 사전 보고 후 회의록에 반영한다.

### 5. QA 및 개발자 확인

- 개발 완료 후 **QA를 반드시 진행한다.**
- QA는 회의록에 명시된 수용 기준(Acceptance Criteria) 전항목을 대상으로 한다.
- **QA 완료 = 수용 기준 전항목 통과 + 담당 개발자(FE 또는 BE) 승인.**
- 수용 기준 없이 QA를 진행하는 것은 허용하지 않는다.

### 6. 스프린트 완료 기준

- QA 완료 후 **액션아이템 완료 체크박스를 업데이트한 시점**을 스프린트 완료로 본다.
- 완료율 기준: **100% 원칙, 불가피한 경우 90% 완료까지 허용.**
- 90% 미만 완료는 스프린트 완료로 인정하지 않으며, 미완료 항목은 다음 스프린트로 이월한다.

### 7. 보고 프레젠테이션

- 스프린트 완료(90% 이상) 확인 후 **PM(Jordan)이 보고용 프레젠테이션을 작성**한다.
- 작성 도구: `/ppt-report` 스킬 사용.

## 완료 기준 (90% 룰)

```
전체 액션아이템 완료율 = 완료된 항목 수 / 전체 항목 수 × 100

- 100%: 스프린트 완료
- 90% 이상 ~ 100% 미만: 불가피한 경우에 한해 스프린트 완료로 인정
  - 미완료 항목은 다음 스프린트 회의 안건으로 자동 이월
  - 이월 사유를 회의록 비고에 명시
- 90% 미만: 스프린트 미완료, 다음 스프린트 착수 전 PM과 원인 검토
```

## 수용 기준(Acceptance Criteria) 작성 규칙

- 회의록의 각 안건 논의 내용 또는 액션아이템 섹션에 체크리스트 형식으로 명시한다.
- PM이 회의 시 최종 확인하고 회의록에 반영한다.
- 기준이 명시되지 않은 항목은 QA 대상에 포함되지 않는다.
- 형식: `- [ ] 검증 내용 (확인 주체)`
