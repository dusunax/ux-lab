## 1. 현재 프롬프트 원문

```
당신은 따뜻하고 공감 능력이 뛰어난 감정 동반자입니다. 사용자가 오늘 있었던 일을 공유하면:

1. 다음 감정 목록에서 주된 감정 하나를 파악하세요: 기쁨, 설렘, 뿌듯함, 슬픔, 외로움, 스트레스, 분노, 억울함, 불안, 혼란, 피로, 무기력함, 평온
2. 상대방의 감정을 인정하고 공감하는 2~3문장의 한국어 응답을 작성하세요. 절대 조언하거나 해결책을 제시하지 마세요. 그냥 함께 있어 주세요.

규칙:
- JSON 객체만 응답하세요. 마크다운, 설명, 감싸는 텍스트 없이.
- JSON은 정확히 두 개의 키를 가져야 합니다: "emotion" (string)과 "empathy" (string).
- "empathy" 값은 한국어로, 따뜻하고 대화체여야 합니다. 딱딱하거나 임상적이지 않게.
- 입력이 모호하거나 감정적으로 중립적이면 기본 감정을 "평온"으로 설정하세요.
- empathy는 150자 이내로 작성하세요.

응답 형식 (엄격히):
{"emotion": "스트레스", "empathy": "오늘 정말 많이 지치셨겠어요. 그 무게를 혼자 감당해왔다는 게 느껴져서 마음이 쓰여요. 수고 많으셨어요."}
```

## 2. 강점 분석

### 2-1. 역할 정의가 감정적으로 명확하다
"따뜻하고 공감 능력이 뛰어난 감정 동반자"라는 페르소나 설정이 간결하고 목적에 부합한다. 모델이 조언자나 분석가가 아닌 동반자임을 명시해 응답 톤을 일관되게 제어한다.

### 2-2. 조언 금지 규칙이 명시적이다
"절대 조언하거나 해결책을 제시하지 마세요. 그냥 함께 있어 주세요." — 이 한 문장이 공감 일기 앱의 핵심 UX 원칙을 정확히 포착한다. 많은 모델이 기본적으로 해결책을 제시하려는 경향이 있으므로 이 명시적 제약은 중요하다.

### 2-3. 감정 목록이 고정되어 있다
13개 감정 카테고리를 열거형으로 제한해 출력 다양성을 제어한다. `normalizeEmotion()` 함수와 함께 작동해 파싱 실패를 줄이는 방어적 설계다.

### 2-4. few-shot 예시가 포함되어 있다
응답 형식 예시 한 개가 JSON 구조와 어조를 동시에 시연한다. 모델에게 "이런 텍스트를 써라"를 보여주는 것이 지시만 하는 것보다 효과적이다.

### 2-5. 출력 형식 제약이 명확하다
마크다운 금지, 키 두 개만 허용, 150자 제한 — 파싱 안정성을 위한 제약이 충분히 구체적이다.

## 3. 개선 가능 영역

### 3-1. few-shot 예시가 단일 감정(스트레스)에 편향되어 있다

**현상:** 예시가 "스트레스" 하나뿐이다.

**문제:** 모델은 few-shot 예시의 감정 톤을 암묵적으로 학습한다. 스트레스 예시만 있으면 밝은 감정(기쁨, 설렘)에서 응답 어조가 과하게 무겁거나 위로 중심으로 치우칠 수 있다.

**권장 조치:** 긍정 감정(기쁨), 부정 감정(슬픔), 중립 감정(평온) 예시를 각각 추가해 감정 스펙트럼 전체를 커버한다.

---

### 3-2. 복합 감정 처리 지침이 없다

**현상:** "주된 감정 하나를 파악하세요"만 있고, 복합 감정 입력에 대한 결정 기준이 없다.

**문제:** 실제 일기는 "오늘 발표는 성공했는데 친구랑 싸웠어요" 같은 복합 감정이 흔하다. 모델마다 우선순위 결정 방식이 달라 fallback 모델 교체 시 감정 분류가 불일치할 수 있다.

**권장 조치:** "복합 감정인 경우, 사용자가 더 많이 서술한 감정 또는 마지막에 언급한 감정을 우선하세요" 규칙 추가.

---

### 3-3. empathy 응답에 호칭 일관성 기준이 없다

**현상:** 현재 예시는 "~셨겠어요" 경어체를 사용하지만, 호칭(당신/니/본인)이나 반말/존댓말 전환 기준이 없다.

**문제:** 모델에 따라 "당신은 오늘 힘드셨겠네요"와 "오늘 많이 힘들었겠다" 중 하나를 임의로 선택해 일관성이 깨질 수 있다. 특히 free tier fallback 모델들은 어조 일관성이 낮다.

**권장 조치:** "항상 존댓말(~요/~습니다)을 사용하고, 호칭을 생략하거나 '오늘'로 시작하세요" 명시.

---

### 3-4. empathy 길이 상한만 있고 하한이 없다

**현상:** 150자 이하 제약만 있다.

**문제:** 일부 모델이 매우 짧은 응답(예: "힘드셨겠어요.")을 반환할 수 있다. 단일 문장은 공감으로 느껴지기보다 형식적으로 보인다. 지시에 명시된 "2~3문장"은 step 2에만 있고 규칙 섹션에서 반복되지 않는다.

**권장 조치:** 규칙 섹션에 "empathy는 2~3문장, 50자 이상 150자 이하로 작성하세요" 추가.

---

### 3-5. 입력 검증 지침이 없다

**현상:** 일기가 아닌 입력(숫자만, 이모지만, 외국어 등)에 대한 처리 지침이 없다.

**문제:** "ㅋㅋㅋ" 또는 "hello" 같은 입력에 모델이 감정을 억지로 분류하거나 부적절한 공감을 생성할 수 있다. `MAX_INPUT_LENGTH = 500` 제한은 있지만 최소 의미 있는 입력 기준이 없다.

**권장 조치:** "입력이 의미 있는 문장이 아닌 경우(5자 미만, 숫자/기호만) emotion을 '평온'으로, empathy를 '오늘은 어떤 하루였나요? 천천히 마음을 나눠주세요.'로 응답하세요" 추가.

## 4. 개선안 — 수정된 프롬프트 제안

```
당신은 따뜻하고 공감 능력이 뛰어난 감정 동반자입니다. 사용자가 오늘 있었던 일을 공유하면:

1. 다음 감정 목록에서 주된 감정 하나를 파악하세요:
   기쁨, 설렘, 뿌듯함, 슬픔, 외로움, 스트레스, 분노, 억울함, 불안, 혼란, 피로, 무기력함, 평온
   - 복합 감정인 경우, 사용자가 더 길게 서술하거나 마지막에 언급한 감정을 우선하세요.
   - 입력이 5자 미만이거나 의미 있는 문장이 아니면 "평온"을 사용하세요.

2. 상대방의 감정을 인정하고 공감하는 2~3문장의 한국어 응답을 작성하세요.
   절대 조언하거나 해결책을 제시하지 마세요. 그냥 함께 있어 주세요.

규칙:
- JSON 객체만 응답하세요. 마크다운, 설명, 감싸는 텍스트 없이.
- JSON은 정확히 두 개의 키를 가져야 합니다: "emotion" (string)과 "empathy" (string).
- "empathy" 값은 한국어 존댓말(~요)로, 따뜻하고 대화체여야 합니다. 호칭은 생략하거나 "오늘"로 시작하세요.
- empathy는 2~3문장, 50자 이상 150자 이하로 작성하세요.
- 입력이 모호하거나 감정적으로 중립적이면 기본 감정을 "평온"으로 설정하세요.

응답 형식 (엄격히, 아래 세 예시를 참고):
{"emotion": "기쁨", "empathy": "오늘 정말 기분 좋은 일이 있으셨군요. 그 설레는 마음이 느껴져서 저도 같이 기뻐요. 오늘 하루 충분히 즐기셨으면 해요."}
{"emotion": "슬픔", "empathy": "많이 속상하셨겠어요. 그 마음이 충분히 이해가 돼요. 오늘은 그냥 그 감정을 느끼셔도 괜찮아요."}
{"emotion": "스트레스", "empathy": "오늘 정말 많이 지치셨겠어요. 그 무게를 혼자 감당해왔다는 게 느껴져서 마음이 쓰여요. 수고 많으셨어요."}
```

### 변경 요약

| 항목 | 변경 전 | 변경 후 |
|------|---------|---------|
| few-shot 예시 수 | 1개 (스트레스) | 3개 (기쁨/슬픔/스트레스) |
| 복합 감정 처리 | 없음 | 서술량/마지막 언급 우선 |
| 호칭/경어 기준 | 없음 | 존댓말, 호칭 생략 명시 |
| 길이 하한 | 없음 | 50자 이상 명시 |
| 짧은 입력 처리 | 없음 | 5자 미만 = 평온 처리 |

## 5. 적용 전 승인 절차

프롬프트 변경은 사용자 대면 응답 품질에 직접 영향을 미치므로 다음 절차를 준수한다.

```
1. Sage (AI)  — 프롬프트 초안 작성 및 baseline-samples.md 기준 자체 검증
2. Jordan (PM) — 감정 분류 정확도 및 사용자 경험 방향성 승인
3. Riley (UX)  — 공감 텍스트 어조, 언어 품질 승인
4. Avery (FE)  — index.html SYSTEM_PROMPT 상수 교체 및 배포
```

- 승인 없이 프롬프트를 변경하거나 배포하지 않는다.
- 변경 시 `baseline-samples.md`에 새 샘플을 추가해 회귀 여부를 확인한다.
- 프롬프트 버전은 이 파일명 suffix로 관리한다 (`prompt-review-v2.md` 등).

## 6. 모델 선택 근거

`api/chat.js`의 FALLBACKS 목록을 기준으로 분석한다.

### 6-1. 현재 모델 목록

```javascript
const FALLBACKS = [
  'openai/gpt-oss-120b:free',
  'meta-llama/llama-3.3-70b-instruct:free',
  'nvidia/nemotron-3-super-120b-a12b:free',
  'openai/gpt-oss-20b:free',
  'qwen/qwen3-next-80b-a3b-instruct:free',
  'minimax/minimax-m2.5:free',
];
```

모든 모델이 `:free` tier를 사용한다. 비용 제로를 유지하면서 가용성을 확보하기 위한 전략적 선택이다.

### 6-2. 각 모델의 특성과 선택 근거

| 순서 | 모델 | 파라미터 | 특성 | 선택 이유 |
|------|------|---------|------|----------|
| 1 | `openai/gpt-oss-120b:free` | 120B | GPT 아키텍처, 고품질 한국어 | 기본 첫 번째 시도. 한국어 지시 수행과 JSON 출력 신뢰도가 높음 |
| 2 | `meta-llama/llama-3.3-70b-instruct:free` | 70B | Instruct 파인튜닝, 빠른 응답 | 1번 rate limit 시 빠른 fallback. Llama 3.3은 다국어 성능 검증됨 |
| 3 | `nvidia/nemotron-3-super-120b-a12b:free` | 120B (MoE A12B) | 대용량 추론 | 복잡한 감정 문맥 처리를 위한 대형 모델 백업 |
| 4 | `openai/gpt-oss-20b:free` | 20B | 경량 GPT 계열 | 빠른 응답 우선 시 후순위 옵션. 가용성 높음 |
| 5 | `qwen/qwen3-next-80b-a3b-instruct:free` | 80B | Qwen3, 한중일 특화 | 한국어 감정 표현에 강한 동아시아 언어 모델 |
| 6 | `minimax/minimax-m2.5:free` | - | 마지막 보루 | 앞선 5개 모두 rate limit 시 최후 fallback |

### 6-3. Fallback 순서 설계 근거

현재 fallback 전략은 **대형 모델 우선, 경량 모델 후순위** 패턴이다.

```
높은 품질 (120B) → 빠른 instruct (70B) → 대형 MoE (120B/A12B) 
→ 경량 (20B) → 한국어 특화 (80B) → 최후 보루
```

이 설계의 근거:

1. **첫 시도에서 최대 품질 확보**: 감정 공감 응답은 창의적 언어 품질이 중요하므로 가장 성능 좋은 모델을 먼저 시도한다.
2. **rate limit이 주된 실패 원인**: `:free` tier는 RPM(분당 요청 수)과 일일 한도가 있어 429가 주요 fallback 트리거다. 모든 후보가 다른 provider이므로 동시 rate limit 가능성이 낮다.
3. **다양한 provider 분산**: OpenAI, Meta, NVIDIA, Qwen, MiniMax — provider를 분산해 단일 장애점을 방지한다.

### 6-4. 개선 제안

- **Qwen 모델 순서 상향 검토**: 한국어 감정 일기 특성상 한중일 특화 Qwen 계열이 3~4번으로 올라와야 할 수 있다. baseline-samples 결과를 비교한 후 판단.
- **DeepSeek 모델 추가 검토**: `deepseek/deepseek-chat:free`가 OpenRouter에서 제공될 경우, 한국어 품질과 JSON 준수율을 검증해 목록에 추가하는 것을 권장.
- **모델별 응답 품질 로깅**: 현재 `chat_success` 로그에 `model_used`가 기록되므로, 향후 감정 분류 정확도를 모델별로 집계하면 순서 최적화가 가능하다.
