## 논의 내용

### 안건 1 — `/api/log` Rate Limit

**현황:** `/api/log`는 공개 POST 엔드포인트다. 이벤트 및 파라미터 키 allowlist로 로그 인젝션은 방어하고 있으나, IP 기반 요청 횟수 제한이 없어 비용 및 로그 노이즈 위험이 있다.

**결정:** Chase가 Vercel Firewall에서 `/api/log`에 대해 IP당 분당 30회 rate limit 규칙을 추가한다. 임계값은 정상 사용 패턴(일기 제출 1회 = 최대 3 이벤트)을 기준으로 설정. 초과 시 429 반환, 클라이언트는 sendBeacon 특성상 오류를 무시한다.

**수용 기준:**
- [ ] Vercel Firewall에 `/api/log` rate limit 규칙 추가 확인
- [ ] 규칙 적용 후 정상 흐름(일기 제출)에서 차단 없음 확인

### 안건 2 — Firebase Analytics `measurementId` 연결

**현황:** `firebaseConfig`에 `measurementId`가 없어 Firebase Analytics SDK가 비활성 상태다. `logEvent()`는 현재 `/api/log` → Vercel Runtime Logs로만 전송되고 있다.

**결정:** Blake가 Firebase Console에서 Google Analytics를 프로젝트에 연결하고 `measurementId`를 환경 변수로 추가한다. Avery가 클라이언트 설정에 반영한다.

**수용 기준:**
- [ ] Firebase Console에서 GA4 연결 및 `measurementId` 확보
- [ ] Vercel 환경 변수에 `NEXT_PUBLIC_FIREBASE_MEASUREMENT_ID` 추가 (또는 동등한 방식)
- [ ] Firebase Analytics `logEvent` SDK 초기화 연동 확인

**현황:** `entry_load_failure`, `entry_save_failure`, `entry_delete`는 allowlist에 등록됐으나 `index.html`에 실제 호출 지점이 없다.

### 안건 4 — `x-request-id` 전달 방식

**논의:** 현재 서버는 `x-request-id`를 응답 헤더로 반환한다. 클라이언트가 이를 읽어 이후 이벤트에 연결하려면 `fetch` 응답 헤더를 직접 파싱해야 한다.

**두 방식 비교:**

| 방식 | 장점 | 단점 |
|------|------|------|
| 응답 헤더 | HTTP 관례에 맞음, body 구조 불변 | 클라이언트가 `res.headers.get()` 코드 추가 필요 |
| JSON body | 클라이언트 구현 단순, 기존 구조 파악 용이 | body schema 변경, 일부 클라이언트 혼동 가능 |

**결정:** **JSON body에 포함**한다. 이유: 이 앱은 싱글 HTML 파일 기반이며, body 파싱 코드는 이미 있다. 헤더 파싱 코드를 별도로 추가하는 것보다 일관성이 높다. 향후 fetch wrapper 분리 시 재검토.

Avery가 클라이언트에서 `result.request_id`를 읽어 `entry_submit_success` 이벤트에 `request_id` 파라미터를 추가한다. Blake가 서버 응답 body에 `request_id` 필드를 포함한다.

`api/log.js` allowlist에 `request_id` 파라미터 키 추가 필요.

**수용 기준:**
- [ ] `api/chat.js` 응답 body에 `request_id` 포함 확인
- [ ] `api/log.js` ALLOWED_PARAM_KEYS에 `request_id` 추가
- [ ] `entry_submit_success` 로그에 `request_id` 필드 포함 확인

### 안건 7 — 모바일 반응형 기본 대응

**현황:** 앱은 데스크탑 엑셀 그리드 테마로 설계됐고 현재 모바일 대응이 없다. 외부 공유 시 모바일로 첫 접근하는 사용자가 이탈할 가능성이 높다.

**Riley 제안:** 완전한 반응형 재설계는 Out of Scope. 최소한의 안전망(mobile viewport 깨짐 방지)만 이번 스프린트에 포함한다.

**결정:** 모바일 최소 대응 범위는 다음으로 한정:
1. `viewport` meta 태그 확인
2. 가로 스크롤 없이 텍스트 입력 및 로그인 버튼 사용 가능
3. 그리드는 가로 스크롤 허용 (재설계 불필요)

Avery 구현, Riley 검수.

**수용 기준:**
- [ ] 375px width 기준 로그인 화면 및 입력 폼이 가로 잘림 없이 표시됨
- [ ] 그리드 영역 가로 스크롤 동작 확인 (overflow-x: auto)

### 안건 9 — `index.html` 파일 분리 검토

**현황:** `index.html`이 약 2,100줄이다. 프로젝트 규칙상 파일 최대 800줄을 초과한 상태.

**논의:**
- Avery: 싱글 HTML은 Vercel 정적 서빙 구조에서 의도된 설계다. 빌드 툴 없이 `<script type="module">` 분리를 하면 CORS 이슈 및 로컬 개발 마찰이 생긴다.
- Jordan: 파일 크기 규칙은 React/Next.js 기준이며, 이 앱은 vanilla JS + 싱글 파일 의도로 시작됐다. 코딩 스타일 가이드에서 이 앱을 명시적 예외로 처리하거나, 빌드 환경 도입을 결정해야 한다.
- Morgan: 현재 크기에서 버그 추적 및 리뷰가 이미 어렵다. 로직 분리 없이는 QA 비용이 계속 늘어난다.

**결정:** **이번 스프린트 Out of Scope**. 단, 스프린트 5 이전에 Jordan이 빌드 환경 도입(Vite 등) 여부를 결정하고 회의 안건으로 올린다. Avery가 분리 가능한 영역(Firebase 초기화, logEvent 유틸, 상수)을 사전에 파악해 제안 자료로 준비.
