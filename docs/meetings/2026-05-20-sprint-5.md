## 논의 내용

### 안건 1 — 빌드 환경 도입 여부

**현황:** `index.html`이 약 2,100줄이며, FE가 스프린트 4에서 분리 가능 영역을 사전 조사 완료했다. 빌드 툴 없이도 CSS 및 일부 JS 분리가 가능하지만, Vite 도입 시 TypeScript + Firebase Modular SDK + 완전한 모듈 분리까지 일괄 처리할 수 있다.

**각 역할 의견:**

- **FE:** Vite 도입은 중장기적으로 맞는 방향이지만, 지금 당장 도입하면 설정 비용, 로컬 개발 환경 재정비, Firebase Modular SDK 전환까지 스프린트 하나를 온전히 소모한다. CSS 분리만 먼저 하면 이번 스프린트 안에 index.html을 ~1,100줄로 줄일 수 있다.
- **BE:** 빌드 환경이 없어도 서버 API 구조와 무관하다. FE 결정을 따른다.
- **SRE:** Vite 도입 시 Vercel 빌드 설정을 변경해야 하고, 현재 정적 서빙 파이프라인이 바뀐다. 리스크가 낮지 않다. 지금은 유지가 낫다.
- **QA:** 빌드 환경 없이 파일을 늘리면 테스트 경계가 불명확해진다. 단위 테스트 진입점이 생기는 utils.js 분리를 우선순위로 둔다면 빌드 도구 없이도 QA 비용이 줄어든다.
- **AI:** 빌드 환경 여부는 프롬프트 파이프라인과 무관하다. 어느 방향이든 문제없다.
- **UX:** 사용자 경험에 직접 영향이 없다. FE + SRE 판단을 지지한다.
- **TS:** 결정 이후 문서화 및 `CLAUDE.md` 예외 처리 조항 정리가 필요하다.

**결정:** **이번 스프린트에서 Vite 도입하지 않는다. 싱글 HTML + 빌드 도구 없는 구조를 유지한다.**

근거: Vercel 빌드 파이프라인 변경 리스크, 스프린트 내 완료 가능성, 현재 앱 규모에서 Vite의 이점이 비용을 상쇄하지 못함. Vite 도입 여부는 TypeScript 마이그레이션 필요성이 생기는 시점(또는 기여자가 2인 이상이 되는 시점)에 재검토한다.

**수용 기준:**
- [ ] Vite 관련 설정 파일 없음 (기존 정적 서빙 구조 유지)
- [ ] TS가 `CLAUDE.md`에 "이 앱은 빌드 도구 없는 싱글 파일 구조이며, 800줄 파일 크기 규칙 예외"임을 명시

### 안건 2 — `index.html` 파일 분리 방향 확정

**현황:** 안건 1에서 Vite를 이번 스프린트에 도입하지 않기로 결정했다. FE 사전 분석 결과:

| 분리 대상 | 방식 | 효과 |
|----------|------|------|
| CSS 전체 (`<style>`, ~927줄) | `<link rel="stylesheet">` | index.html 927줄 제거, 리스크 없음 |
| `config.js` | `<script src>` | 프롬프트·감정 설정 격리 |
| `utils.js` | `<script src>` | 순수 함수, QA 단위 테스트 진입점 |
| `api.js`, `storage.js` | `<script src>` | 외부 I/O 격리 |

**논의:**

- **FE:** CSS 분리는 즉시 가능하고 리스크가 없다. JS 분리는 `<script src>` 방식으로 하면 전역 스코프 의존성 문제가 있다. config.js와 utils.js는 분리 가능하지만, api.js와 storage.js는 Firebase 객체를 참조하므로 초기화 순서 문제가 생길 수 있다. 이번 스프린트에서는 CSS + config.js + utils.js까지 분리한다.
- **QA:** utils.js가 분리되면 함수 단위 검증이 가능해진다. 환영한다.
- **SRE:** CSS를 별도 파일로 분리하면 Vercel 정적 서빙에서 별도 요청이 발생하지만 캐싱 효과가 있어 오히려 성능이 나아진다.
- **TS:** 분리 후 파일 목록을 README에 반영해야 한다.

**결정:** **이번 스프린트에서 CSS 분리 + config.js + utils.js 분리를 진행한다. api.js / storage.js 분리는 Out of Scope.**

CSS 분리로 `index.html`이 약 1,100줄로 감소. config.js + utils.js 분리 후 index.html은 최종적으로 **약 800줄 이하** 달성을 목표로 한다.

**수용 기준:**
- [ ] `styles/` 폴더에 CSS 파일 분리, `index.html`에서 `<link rel="stylesheet">` 로드 확인
- [ ] `config.js` 분리 후 프롬프트·감정 상수 정상 동작 확인
- [ ] `utils.js` 분리 후 기존 유틸리티 함수 정상 동작 확인
- [ ] `index.html` 줄 수 800줄 이하 달성
- [ ] 로컬 및 Vercel 배포 환경 모두에서 정상 동작 확인 (QA 검수)

### 안건 3 — 동의 UX 배너: 해외 출시 기준 정의

**현황:** 스프린트 4에서 동의 UX 배너를 "해외 출시 시 즉시 추가"로 결정했으나, 해외 출시의 기준이 정의되지 않았다. 이 기준 없이는 트리거 시점을 식별할 수 없다.

**논의:**

- **UX:** 기준이 모호하면 실제로 배너가 추가되지 않는 리스크가 있다. 사용자 보호 관점에서 가능한 한 구체적인 기준을 정하는 것이 낫다. 단, 현재 앱이 한국어 UI이고 국내 사용자 대상이므로 배너 부재가 당장 법적 문제가 되지는 않는다.
- **BE:** GDPR은 EU 거주자의 데이터를 처리할 때 적용된다. 도메인 변경보다 실제 EU 사용자 접근 여부가 더 정확한 기준이다. GA4에서 국가별 사용자를 확인할 수 있다.
- **SRE:** Vercel Analytics에서 국가별 트래픽을 확인 가능하다. 특정 국가 트래픽이 일정 비율을 넘으면 알림 설정을 고려할 수 있다.
- **QA:** 기준이 정해지면 체크리스트로 만들어 추적한다.
- **TS:** 결정 사항을 Open Questions에서 닫고 Resolved로 처리한다.

**결정:**

해외 출시 기준을 아래와 같이 정의한다:

| 트리거 조건 | 내용 |
|------------|------|
| 조건 A | GA4 기준 비한국 사용자(non-KR)가 **누적 50명** 초과 |
| 조건 B | 외부 공유 대상에 **비한국어권 사용자가 명시적으로 포함**될 때 |
| 조건 C | 앱 UI 언어를 한국어 외 언어로 추가할 때 |

셋 중 하나라도 충족되면 동의 배너 구현을 **최우선 작업(P0)**으로 착수한다. 배너 내용: 로그 수집 사실 고지 + 거부 시 수집 중단(logEvent 비활성화).

**현재 상태:** 세 조건 모두 미충족. **이번 스프린트 Out of Scope 유지.**

**수용 기준:**
- [ ] TS가 위 기준을 Open Questions → Resolved로 처리하고 트리거 조건을 문서화
- [ ] PM이 GA4에서 국가별 사용자 모니터링 주기적 확인 (스프린트 종료 시마다)

### 안건 4 — AI 관여 시점 및 프롬프트 품질 개선

**현황:** 현재 OpenRouter를 통해 LLM이 감정 일기 공감 응답을 생성한다. 프롬프트는 초기 MVP 버전이며, 개선되지 않은 상태다. AI 전담 역할이 아직 관여하지 않았다.

**AI 현황 파악:**

- 현재 시스템 프롬프트: 감정 분석 + 공감 응답 생성 단일 프롬프트
- 모델 선택: OpenRouter 기본 설정 (정확한 모델명 확인 필요)
- 실패 처리: `chat_exhausted` 상태로 오류 UX 처리됨 (Sprint 4 구현)

**AI 의견:** 관여 시점은 "지금"이 맞다. 프롬프트가 바뀌면 응답 품질이 즉각 달라지고, 이는 핵심 UX다. 최소한 두 가지는 이번 스프린트에 다뤄야 한다: (1) 현재 프롬프트 리뷰 및 개선안 작성, (2) 모델 선택 근거 명문화.

**UX 의견:** 사용자 테스트 없이 프롬프트를 바꾸면 기존 경험이 훼손될 수 있다. 변경 전후 응답 샘플 비교가 필요하다.

**BE 의견:** 프롬프트 변경은 코드 변경이 아니므로 배포 없이 환경 변수로 관리하면 리스크를 낮출 수 있다. 프롬프트를 환경 변수로 분리하는 것을 고려해볼 만하다.

**QA 의견:** A/B 테스트 없이 프롬프트를 교체하면 회귀 기준이 없다. 기존 프롬프트 응답 샘플 5개 이상을 기준선으로 보존해야 한다.

**결정:** **이번 스프린트에서 AI 관여를 시작한다. 범위는 다음으로 한정:**

1. AI가 현재 프롬프트를 리뷰하고 개선안을 작성한다 (문서로 제출, FE 반영은 검토 후 결정).
2. 현재 OpenRouter에서 사용 중인 모델명을 확인하고 선택 근거를 문서화한다.
3. 프롬프트 변경 시 응답 샘플 5개 이상을 기준선으로 보존하는 것을 QA와 합의한다.
4. 프롬프트를 Vercel 환경 변수로 분리하는 것은 **Out of Scope** (복잡도 증가 대비 이점 불명확).

**수용 기준:**
- [ ] AI가 현재 시스템 프롬프트 리뷰 문서 작성 (`docs/ai/prompt-review-v1.md`)
- [ ] 현재 사용 모델명 및 선택 근거 문서화
- [ ] 기존 응답 샘플 5개 이상 `docs/ai/baseline-samples.md`에 보존
- [ ] 프롬프트 개선안이 있을 경우 PM + UX 승인 후 FE 반영

### 안건 5 — CORS 제한

**현황:** `api/log.js`와 `api/chat.js`의 `Access-Control-Allow-Origin: *`이 모든 도메인에서 요청을 허용한다. 현재 allowlist로 로그 인젝션은 방어 중이나, 다른 도메인에서 API를 직접 호출할 수 있는 상태다.

**논의:**

- **BE:** `*` 와일드카드는 인증이 붙은 엔드포인트에서는 브라우저가 credentials를 보내지 않아 실질적 보안 문제가 크지 않다. 하지만 `/api/log`는 인증 없는 공개 엔드포인트이므로 도메인 제한이 의미 있다.
- **SRE:** Vercel 도메인은 `ai-empathy-diary.vercel.app`이다. `vercel.app` 서브도메인이나 커스텀 도메인이 추가되면 allowlist를 업데이트해야 하므로 유지보수 비용이 생긴다.
- **FE:** 로컬 개발 시 `localhost:xxxx`에서 `/api/log`를 호출해야 한다. CORS 제한 시 개발 환경 처리가 필요하다.
- **QA:** 변경 후 반드시 로컬 및 프로덕션 양쪽에서 로그 수신 테스트를 해야 한다.

**결정:** **`api/log.js` CORS를 Vercel 프로덕션 도메인으로 제한한다. 개발 환경에서는 `localhost` 허용.**

허용 도메인 목록:
- `https://ai-empathy-diary.vercel.app`
- `http://localhost` (포트 무관, 개발 환경)
- 환경 변수 `ALLOWED_ORIGIN`으로 추가 도메인 지원 (커스텀 도메인 대비)

`api/chat.js`는 Firebase Auth 토큰으로 인증이 이뤄지므로 이번 스프린트에서는 변경하지 않는다.

**수용 기준:**
- [ ] `api/log.js` `Access-Control-Allow-Origin` 값이 `*`에서 명시적 도메인으로 변경됨
- [ ] 프로덕션 배포 후 로그 수신 정상 확인
- [ ] 로컬 개발 환경(`localhost`)에서 CORS 오류 없이 로그 전송 확인
- [ ] QA가 Origin이 다른 도메인에서 요청 시 CORS 오류 반환 확인

### 안건 6 — 모바일 반응형 심화

**현황:** 스프린트 4에서 375px 기준 로그인/입력 폼 잘림 방지를 완료했다. 그리드는 가로 스크롤 허용 상태다. 외부 공유 이후 모바일 이탈률 데이터가 없어 투자 규모를 판단하기 어렵다.

**UX 의견:** 감정 일기는 모바일에서 쓰는 경우가 많다. 현재 그리드가 모바일에서 표 형태로만 존재해 텍스트 입력 → 일기 목록 확인 흐름이 불편하다. 이번 스프린트에서 모바일용 카드 뷰(리스트 형태) 전환을 제안한다.

**FE 의견:** 카드 뷰 전환은 렌더링 로직을 분리해야 하므로 JS 분리(안건 2)와 충돌이 생길 수 있다. 두 작업을 같은 스프린트에 하면 충돌 리스크가 크다.

**PM 판단:** 데이터 없이 모바일 카드 뷰에 스프린트 용량을 투자하는 것은 조기 최적화다. 외부 공유 후 GA4에서 기기 유형별 이탈률을 확인한 뒤 결정한다.

**결정:** **이번 스프린트 Out of Scope. 모바일 카드 뷰는 외부 공유 후 GA4 데이터 확인 시까지 보류.**

단, 스프린트 4에서 완료한 최소 대응(375px, 가로 스크롤)이 배포 환경에서 정상 동작하는지 QA가 재확인한다.

**수용 기준:**
- [ ] 프로덕션 URL에서 375px 기준 로그인/입력 정상 확인 (QA)
- [ ] 향후 재검토 기준: GA4 기기 유형별 이탈률 데이터 2주치 확보 후 UX 제안

### 안건 7 — DAU 50+ 대비 P2 이벤트 추가

**현황:** 현재 P0 이벤트(page_view, entry_submit_success 등)와 P1 이벤트(entry_load_failure 등)가 구현됐다. P2 이벤트(`auth_signout`, 감정 분포 분석, entry_delete 분포)는 DAU 50+ 도달 후 필요성을 판단하기로 했다.

**SRE 의견:** 현재 DAU가 10 미만이다(개발팀 수준). P2 이벤트를 지금 추가하는 것은 낭비다.

**AI 의견:** 감정 분포 이벤트는 프롬프트 품질 개선에 피드백 루프로 활용할 수 있다. DAU 50+가 아니더라도 감정 라벨 분포 데이터는 조기에 수집하는 것이 가치 있다.

**QA 의견:** 이벤트 추가는 allowlist 업데이트와 call site 구현을 동시에 해야 한다. 범위가 크지 않으므로 결정되면 빠르게 할 수 있다.

**PM 판단:** DAU 기준을 유지한다. 단, AI가 프롬프트 개선을 위해 감정 분포 데이터가 필요하다는 논리는 타당하다. 감정 라벨 이벤트 하나만 선별해 P1.5로 추가한다.

**결정:** **P2 이벤트 전체 추가는 DAU 50+ 이후로 보류. 단, `emotion_label_recorded` 이벤트 하나를 이번 스프린트에 추가한다.**

`emotion_label_recorded`: 사용자가 일기를 제출하고 AI 감정 분석 결과를 수신했을 때 발생. 파라미터: `{ label: string }` (감정 라벨명만, 텍스트 원문 금지).

**수용 기준:**
- [ ] `api/log.js` ALLOWED_EVENTS에 `emotion_label_recorded` 추가
- [ ] ALLOWED_PARAM_KEYS에 `label` 추가 (기존 키와 충돌 없음 확인)
- [ ] AI 응답 수신 성공 후 클라이언트에서 `logEvent('emotion_label_recorded', { label })` 호출 확인
