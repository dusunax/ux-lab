# Conflict Resolution Workflow

## 파일 충돌 시 필수 절차

충돌이 발생하면 **즉시 Edit을 실행하지 않는다.** 반드시 아래 순서를 따른다.

### Step 1 — 충돌 원인 설명

사용자에게 아래 항목을 먼저 설명한다:

- 어떤 파일에서, 어떤 변경이 충돌했는지
- 두 hunk의 내용 차이 (내 변경 vs 현재 파일 상태)
- 충돌 원인 (동시 수정, race condition, 직전 edit 실패 후 재시도 등)

### Step 2 — 병합 계획 수립

어떤 hunk를 어떻게 합칠지 계획을 사용자에게 제시한다:

```
병합 계획:
- [hunk A]: 현재 파일 내용 유지 / 내 변경으로 대체 / 두 변경 병합
- [hunk B]: ...
이유: [각 결정의 근거]
```

사용자가 계획을 확인한 뒤 진행한다.
(명시적 승인 없이도 계획이 명확하면 진행 가능하나, 파괴적 변경이면 확인 후 진행)

### Step 3 — 계획에 따라 파일 수정

승인된 계획대로만 Edit한다. 계획에 없는 추가 변경을 끼워 넣지 않는다.

### Step 4 — 관련 테스트 먼저 실행

전체 테스트보다 **충돌 파일과 직접 연관된 테스트만** 먼저 실행한다.

```bash
# 예: 수정된 파일이 api/chat.js라면
npm test -- --testPathPattern=chat

# 예: 빌드 영향 확인만 필요하다면
npm run build 2>&1 | tail -20
```

### Step 5 — 전체 영향 요약 (10줄 이내)

```
충돌 해결 요약
- 파일: [파일명]
- 원인: [한 줄]
- 병합 방식: [선택한 전략]
- 변경 hunk: [n]개
- 테스트: [통과 / 실패 / 해당 없음]
- 사이드 이펙트: [있으면 설명, 없으면 "없음"]
```

---

## 트리거 조건

이 워크플로우를 적용하는 경우:

- `git merge` / `git rebase` 중 충돌 발생
- Edit 시도 시 "File has been modified since read" 오류
- PR merge 시 conflict marker (`<<<<<<<`, `=======`, `>>>>>>>`) 발견
- 두 에이전트가 같은 파일을 동시 수정한 상황

## 예외

단순 오타 수정 또는 주석 변경 등 충돌 영향 범위가 명확히 1줄이고 로직 변경이 없는 경우,
계획 제시 단계를 생략하고 바로 수정 후 요약만 작성한다.
